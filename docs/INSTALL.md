# Installing issueclaw

## Prerequisites

- **Python 3.12+**
- **[uv](https://docs.astral.sh/uv/)** — `curl -LsSf https://astral.sh/uv/install.sh | sh`

## Install the CLI

```bash
curl -fsSL https://raw.githubusercontent.com/aviadr1/issueclaw/main/install.sh | sh
```

Or manually:

```bash
uv tool install git+https://github.com/aviadr1/issueclaw.git
```

Verify: `issueclaw self detect`

Upgrade: `issueclaw self update`

## Environment variables

| Variable | When needed | Description |
|---|---|---|
| `LINEAR_API_KEY` | Always | Linear API key. Generate at https://linear.app/settings/api (format: `lin_api_...`) |
| `LINEAR_WEBHOOK_SECRET` | Webhooks | Signing secret for webhook validation (auto-generated by `issueclaw init`) |

Set in your shell, `.env` file, or as GitHub Actions secrets.

## Quick start — local

```bash
export LINEAR_API_KEY=lin_api_...

# Initialize repo + pull all data
issueclaw init --repo-dir .

# Pull latest from Linear
issueclaw pull

# Check sync status
issueclaw status

# Edit a markdown file, then push to Linear
issueclaw push

# Preview what would change
issueclaw diff

# Create entities directly
issueclaw create issue --team AI --title "Fix login bug" --priority 2
echo "Root cause found." | issueclaw create comment --issue AI-123
```

## Quick start — CI/CD (consumer repo)

### 1. Initialize

```bash
export LINEAR_API_KEY=lin_api_...
issueclaw init --repo-dir /path/to/your-repo --webhook-url https://your-worker.workers.dev
```

This does everything:
- Saves `LINEAR_API_KEY` to `.env` (gitignored)
- Copies workflow files to `.github/workflows/`
- Sets `LINEAR_API_KEY` as a GitHub secret (if `gh` CLI available)
- Registers a Linear webhook (if `--webhook-url` provided)
- Runs initial `issueclaw pull`

### 2. Set GitHub secrets (if not done by init)

In your repo's Settings > Secrets and variables > Actions:

| Secret | Value |
|---|---|
| `LINEAR_API_KEY` | Your Linear API key |

### 3. Set up webhooks (optional, for real-time sync)

Linear webhooks require a publicly accessible endpoint. Deploy the included CloudFlare Worker:

```bash
cd workers/issueclaw-webhook-proxy/
wrangler secret put LINEAR_WEBHOOK_SECRET   # from issueclaw init output
wrangler secret put GITHUB_TOKEN            # GitHub PAT with repo scope
wrangler secret put GITHUB_REPO             # e.g. "myorg/my-linear-repo"
wrangler deploy
```

Without webhooks, the scheduled sync (every 15 min by default) keeps the repo current.

### 4. Ongoing workflow

After init, CI handles everything:

```
Linear change (or scheduled cron)
  |
  v
issueclaw pull                  (fetches Linear GraphQL API)
  |-- incremental: only changed issues since last sync
  |-- writes markdown to linear/teams/{TEAM}/issues/
  |-- updates .sync/id-map.json
  |
  v
git commit + push

---

Git push to main (someone edits a .md file)
  |
  v
issueclaw push                  (diffs markdown vs Linear)
  |-- detects changed frontmatter fields (status, assignee, etc.)
  |-- detects body changes (description)
  |-- detects new comments
  |-- pushes changes to Linear GraphQL API
```

## How sync works

**Pull** (Linear → git): Fetches issues, projects, initiatives, documents via GraphQL. Renders each as a markdown file with YAML frontmatter. Uses `updatedAt` timestamps for incremental sync.

**Push** (git → Linear): Diffs the markdown files against `.sync/` state. Detects changed frontmatter fields and body content. Pushes changes to Linear via GraphQL mutations.

**Conflict resolution**: Git merge. If both sides changed, git handles the conflict. issueclaw doesn't implement custom conflict resolution — it trusts git.

## Repo structure after init

```
your-repo/
  .env                           # LINEAR_API_KEY (gitignored)
  .sync/
    id-map.json                  # file path → Linear UUID mapping
    state.json                   # sync timestamps
  .github/workflows/
    issueclaw-sync.yaml          # scheduled pull (every 15 min)
    issueclaw-webhook.yaml       # real-time webhook pull
    issueclaw-push.yaml          # push on git commit to linear/
  linear/
    teams/{TEAM}/issues/
      {ID}-{slug}.md             # one file per issue
    projects/{name}/
      _project.md                # project metadata
      updates/{date}.md          # status updates
    initiatives/{name}.md
    documents/{title}.md
```

## Development setup

```bash
git clone https://github.com/aviadr1/issueclaw
cd issueclaw
uv sync                          # install deps
pre-commit install               # install git hooks

# Run tests
uv run pytest

# Type check
uv run pyright

# Lint
uv run ruff check && uv run ruff format --check
```

## Not on PyPI (yet)

issueclaw is installed from GitHub. Consumer repos don't list it in `pyproject.toml` — CI installs it fresh each run.
